MCP & the Machine-Readable Interface

The Model Context Protocol is the connective tissue between a design system and an agent — and how you structure what it exposes makes or breaks accuracy and cost.

What MCP does

The Model Context Protocol is a standard for feeding AI tools the real, structured data behind a design — its intent, tokens, variables, and components — rather than letting them guess from screenshots.1 Figma's Dev Mode MCP server turns a Figma file into an interactive context layer that AI coding tools can query: "X-ray vision" for spacing, color, and component logic, enabling higher-fidelity code generation, especially when the system is cleanly structured.1

Storybook MCP as connective tissue

In the agentic-2026 session, the Storybook MCP server is demonstrated as the connective tissue that exposes design-system components, APIs, and interaction tests to AI agents in a structured, token-optimized format.2 The system's components, not generic web knowledge, become the agent's vocabulary.

JSON for MCP, Markdown for LLM

The most important structural lesson: early attempts dumped Markdown docs into an MCP server, causing high token usage, hallucinations, and poor adherence to component contracts. Encoding component APIs and variants as structured JSON instead reduced token usage by ~80% (per Indeed's case study) and dramatically improved accuracy and cost.3

The emergent pattern is "JSON for MCP, Markdown for LLM": machine-readable structured data (props, variants, allowed values) as JSON schemas; human-oriented guidance and interaction rules as Markdown. Metadata stays under source control with a pipeline that converts MDX/Storybook docs into MCP-readable JSON whenever components change.3 Atlassian implemented this pattern by distributing modular instruction blocks (types, examples, guidelines) to their MCP server (@atlaskit/ads-mcp) and other agents using a structured offerings.json schema (see Agentic Content & Prototyping Templates).

Expose foundations, not just components

A specific failure mode: MCP interfaces that reveal components but omit foundations (spacing, typography) force agents to guess and break brand consistency.3 This is the practical cash-value of the tokens-as-API idea — the foundation layer must be machine-visible too.

Structured queries and APIs

The concept of exposing system schemas programmatically is not new; Shopify's Polaris design system previously pioneered this by offering a GraphQL API to expose components, tokens, pages, and examples directly to consumers.4 In an agentic workflow, this query-based structure becomes the interface protocol. By querying the design system as a structured database or knowledge graph rather than traversing human docs, agents can programmatically extract precise context (e.g., retrieving only components matching icons AND onboarding), optimizing performance and accuracy.4 See Design Systems as Knowledge Graphs.

Perspectives: structured data vs. prose

The corpus is unusually unanimous here: machine interfaces want structure, humans want prose, and conflating them is the original sin of design-system MCP work.3 The Figma Dev Mode MCP and Storybook MCP are both instances of the same move — give the agent the structured truth (tokens, variants, tests) rather than a rendered picture or a wall of docs.12 See AI-Assisted Authoring for MCP in a real authoring loop.
