Design Systems as Knowledge Graphs

Deconstructing design systems as non-linear hyperobjects composed of transcluded content, bidirectional links, unresolved fragments, and structured queries.

Beyond the product metaphor

While design systems are commonly treated as standalone software products, libraries, or documentation sites, Chase McCoy argues they represent something far more complex. Good systems are hyperobjects that capture decisions, language, patterns, history, and the collective institutional knowledge of an organization.1 Instead of flat, linear files, a design system resembles a knowledge graph—a collection of interlinked descriptions of entities, concepts, and objects.1

The four graph mechanics

McCoy identifies four core patterns derived from knowledge graph software (such as Roam Research, Obsidian, and Notion) that can be directly applied to design system architecture:1

  • Transclusion: Storing information (styles, tokens, guidelines, properties) in its minimum viable format so it can be referenced and dynamically embedded across multiple contexts.1 Future updates propagate automatically, preventing documentation drift. Real-world implementations include:
    • Seeds design system: Embeddable, hoverable tags for tokens, components, and properties that show inline details without switching context.1
    • Kiwi Orbit: Inline transclusion tags displaying component metadata directly inside documentation prose.1
  • Bidirectional links: Making components and documents aware of where they are referenced.1 For instance, if a Button component consumes an Icon, the documentation for Button shows it imports Icon, and the documentation for Icon dynamically lists Button as a consumer. This relationship can be visualized as a visual graph or navigated using layered sliding cards (similar to Andy Matuschak's sliding pane interface).1
  • Fragments (Red Links): Creating placeholder entities in the graph that do not yet exist, inspired by Wikipedia's "red links."1 These fragments prevent new pages from being orphaned, establish a clear backlog of design system needs, and lower the friction for community-driven contributions.1
  • Queries: Exposing system content to logical search operators (e.g., query for icons AND onboarding to find onboarding-related assets) and transcluding the results dynamically.1 Shopify's Polaris design system implements a version of this by exposing a GraphQL API of its tokens, components, pages, and examples.1

The Agentic Frontier: Ideal for LLM consumption

While originally proposed in 2021 to improve human authoring and navigation, McCoy's knowledge-graph framework aligns precisely with the requirements of **Agentic AI** and LLM-assisted file management. By converting implicit code relationships into explicit graph edges, the graph structure resolves the major limitations of traditional file management:23

Knowledge Graph Mechanic LLM Context Optimization Agentic Failure Resolved
Transclusion Generates precise, decoupled "context bubbles" (e.g., passing only the spacing token subset to the prompt).2 Resolves Monolithic component definitions; prevents token bloat and hallucination.2
Bidirectional Links Allows the orchestration layer to trace downstream dependencies and run targeted tests when code is changed.2 Resolves Documentation drift; ensures changes don't silently break consuming layers.2
Fragments Acts as an explicit, machine-readable **task backlog** for autonomous agents.2 Prevents Orphaned assets; guides agents to fill identified system gaps safely.2
Queries Allows agents to query data structures programmatically ("JSON for MCP, Markdown for LLM").3 Resolves Unstructured docs in MCP; provides clean schemas instead of bloated prose.3

Perspectives: Linear documents vs. Networked graphs

A central tension exists in how we structure design system knowledge:

  • Linear, product-centric documentation (Traditional): Documentation modeled after standard software libraries—long, top-to-bottom documents. While easy for humans to read sequentially, it hides relationships, duplicates information, and is highly prone to drift as the codebase evolves.1
  • Networked, graph-centric documentation (McCoy): Documentation modeled as a web of addressable, transcluded nodes. While structurally resilient, keeping it human-friendly requires custom UI layers (such as transclusion hovers or sliding panels).1

In the context of modern development, this debate is resolved by maintaining a **dual-interface system**: exposing human-friendly markdown interfaces for designers and developers, while mapping the underlying system metadata into a structured, queryable JSON knowledge graph (e.g., via MCP servers) for machine agents.3
