Three-Tier Token Architecture

A layered token model that separates raw values from meaning from component hooks — the seam that unlocks multi-brand, multi-theme flexibility.

Tokens replace the brand book

Frost frames design tokens as the replacement for the static brand style guide: instead of every team manually interpreting a PDF, all products connect to a token system so a single value remap propagates everywhere automatically, avoiding painful, duplicative manual rebranding across codebases.1

The three tiers

TierNameExampleAudience
1Primitives (definitions)starbucks-green: #00704ASystem owners only — "cornstarch behind the double doors." Not exposed to consumers.
2Semantic tokensbackground-brand: starbucks-greenMost teams consume these — they assign jobs/meaning to primitives.
3Component tokens (sparingly)button-backgroundReserved for genuine multi-brand differentiation.

Tier 1 defines what's available; Tier 2 assigns meaning; Tier 3 is fine-grained component hooks used only when Tier 2 alone cannot solve the problem.1

Themes as collections of token decisions

A theme is not a one-off — it is the set of token values hung together for a context. An organization may run 1 theme (simple), 2 (light + dark), 4 (multiple products), or 8+ (multiple brands / white-label clients), and the architecture should swap the appropriate theme per context.1 Dark mode in particular should be implemented by overriding only the color tokens (typography and spacing usually stay), not by duplicating stylesheets.1

Separate structure from aesthetics

Frost reframes the old HTML/CSS/JS split as a new separation of concerns: a structural layer (the door frame), a behavior layer (JS/component system), and a token layer (paint and hardware — purely aesthetic). Keeping the aesthetic layer isolated from structure is precisely what enables multi-theme, multi-brand, multi-platform flexibility.1

Tokens as the API for the design language

The strongest claim: treat tokens not merely as CSS variables but as the contract between the design system and every consumer — web, native, CMS, QA automation, and AI tooling alike.1 This "tokens-as-API" framing is exactly why tokens matter for agentic readiness: well-named tokens are machine-consumable foundations, and AI workflows depend on agents seeing them.2

Perspectives: how many tokens is too many?

A consistent anti-pattern warning runs through the sources: do not create a token for every CSS property on every component. Systems with thousands of component tokens become unmaintainable.1 The agentic literature adds a related failure mode — monolithic component definitions that bundle foundations, style, and behavior make it hard for both humans and agents to reason about any one of them.2 The shared lesson: let Tier 2 do the heavy lifting; reach for Tier 3 only for genuinely gnarly brand differentiation.

Flexibility, not uniformity

Frost pushes back on the criticism that design systems make everything look the same. A well-architected token system — structure separated from aesthetics — is what lets each brand look different while sharing one component library. The door frame stays the same; the paint and hardware can be unique per brand.1 This claim sits in direct dialogue with Dan Mall's argument that consistency is a poor goal — see Success Metrics.
