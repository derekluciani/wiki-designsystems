Agentic Readiness & Architecture

If your design system isn't structured for machines, agents will ignore your patterns and invent their own — readiness is the work of becoming machine-readable infrastructure.

The shift to agentic workflows

Brad Frost and Dominic Nguyen's thesis: autonomous agents are now assembling UIs from component libraries, and if a design system isn't properly structured, agents will ignore the patterns and create their own, producing inconsistent output.1 Design systems must evolve into machine-readable infrastructure to stay relevant. An "agentic design system" is infrastructure that lets agents autonomously read, reason over, and build with components, tokens, and guidelines — with guardrails so output stays on-brand and accessible.2

Itai Vonshak offers a different starting point: design systems as currently constructed are "dinosaurs in the age of AI."3 While AI is transforming code generation, static design system libraries remain rigid structures that slow development down. Rather than just making static rulebooks machine-readable, Vonshak argues that the real agentic opportunity is to use AI to create user experiences that are consistent when needed, but also truly dynamic, intelligent, and adaptable.3 This shifts the goal of an agentic system from automating compliance to enabling responsive, context-aware design.

The dual pillars: coverage and validation

Frost names two areas teams must invest in to get production-ready agent output:1

  • Coverage — clear examples, metadata, types, and constraints so agents understand how to use components correctly. (Thorough descriptions and JSDoc comments are essential for agents to interpret a library's intent.)
  • Validation — robust functional, accessibility, and visual tests plus human-in-the-loop sign-offs, acting as guardrails against errors. Tests are the "sauce" that gives confidence in what agents assemble.

Three elements of the architecture

The report describes a recurring shape: always-on design foundations, on-demand component metadata via APIs/MCP servers, and orchestration rules governing what agents can do.2 Control flow is usually workflow-oriented (clearly defined steps with limited autonomy per step) rather than a single free-running agent: client → orchestration layer → tools (MCP server, Figma API, token service, CI, Storybook) → target repos.2

Layered architecture for "context bubbles"

Large product companies have restructured systems into independent layers — foundations (tokens/primitives), visual style, and behavioral logic — so agents can reason about small "context bubbles" (just spacing tokens, just button styles) instead of parsing monolithic component definitions.2 This is the same decomposition principle from systems thinking, now serving machines as well as humans. However, this file-splitting approach presents a practical tension with tool-specific limits, where consolidating guidelines into a single file with priority headers is sometimes favored to prevent context-parsing failures (see Agentic Content & Prototyping Templates).

Failure modes specific to design systems

Five recurring failures when connecting agents to design systems:2

  1. Documentation drift — docs, tokens, and code disagree, so agents pick the wrong source of truth.
  2. Unstructured docs in MCP — dumping longform Markdown into machine-consumed interfaces produces bloated prompts and hallucinations.
  3. No explicit trust levels — agents make high-impact changes (merging PRs, editing tokens) without safeguards.
  4. Foundations not visible — MCP reveals components but omits spacing/typography, so agents guess and break brand consistency.
  5. Monolithic component definitions — single huge docs make it hard to reason about foundations, style, and behavior separately.

Teams fix these by aligning docs with actual components, separating human- and machine-oriented representations, and layering foundations/style/behavior.2

Redefining roles

Frost argues design-ops and engineering should focus on building the "constraint system" and "quality pipelines" rather than individual components — and that this shift lets non-engineers (PMs, sales leads) effectively "code" by prompting agents, while engineers retain control through the system's rigor.1 See AI-Assisted Authoring and Evals & Observability. This transition exemplifies the cooperative compounding of human capital (design-ops setting guidelines) and token capital (automated code generation). See Model Sovereignty & Token Capital.

Perspectives: readiness as prerequisite

A throughline worth holding onto: every agentic capability in this corpus is downstream of non-AI rigor. Clean tokens (tokens-as-API), aligned docs, layered architecture, and real tests are the prerequisites; agents amplify whatever foundation they're given — including its flaws. "Your design system is not ready for AI agents" is less a warning about AI than about the state of the foundations.2
