Agentic Content & Prototyping Templates

Managing agentic content as structured blocks, leveraging hybrid code templates, and using configuration files to prevent code-generation hallucinations.

Introduction

Atlassian's case study on AI prototyping demonstrates that autonomous models can effectively utilize design systems for rapid prototyping when grounded in structured rules and templates. By pairing design system rules with preconfigured code, they achieved about 70% component accuracy in a single pass from an image screenshot, which continuously improved over iterations.1 This workflow shifts the design-engineering interface from handoffs to active "handshakes."1

What is Agentic Content?

Atlassian defines Agentic Content as "the practical, plain‑language instructions, examples, and constraints we feed to agents and LLMs so they know what to do and how to respond."1 Under this definition, instructions are no longer simple static prompts but are treated as a structured, maintainable design system asset. Key practices include:1

  • Granular Building Blocks: Breaking prompts and documentation down into structured segments, including guidance, code examples, TypeScript types, keywords, and metadata.
  • Offerings Schema: Defining a schema (like Atlassian's offerings.json) that routes package-specific instructions to the appropriate tools, keeping prompt context synchronized.
  • Tool Integration: Routing this structured content to power a dozen core tools, including MCP servers (such as @atlaskit/ads-mcp), prototyping tools, AI-powered code editors, and custom search/research agents (like Rovo).

Perspectives: Consolidated Guidelines vs. Context Bubbles

A key tension emerged when deciding how to structure design-system instructions for AI agents:1

  • The Context Bubble Approach: As outlined in Agentic Readiness, many teams split instructions into separate, isolated files for tokens, buttons, and typography. This restricts token consumption and helps agents focus on isolated context.
  • The Consolidated Guidelines Approach: Atlassian found that most AI prototyping tools struggled to parse multiple files for context. To resolve this, they consolidated button, typography, and token guidelines into a single guidelines.md file (~2,000 lines).

To prevent this monolithic file from causing prompt bloat or hallucinated details, Atlassian adopted a vendor recommendation: placing a Table of Contents (TOC) with high-priority rules at the very top. This priority-header structure optimized generation speeds and significantly reduced hallucinated component imports or API usage.1

Structured Configuration over Code Generation

A common failure mode in AI code generation is the rewriting of established component code for trivial design variations. For instance, prompting an agent to "change the top-left navigation logo to Confluence" often causes the model to rewrite navigation logic, revert to obsolete component APIs, or break imports.1

Atlassian resolved this by shifting UI choices into a JSON configuration file. The AI model edits only the JSON configuration, leaving the underlying React component code untouched. This constraints-based approach dropped logo-related hallucinations and API regressions to near zero, proving that structured configurations are more deterministic and reliable than open-ended code-generation prompts.1

Hybrid Templates and Figma Inheritance

Relying solely on text instructions was insufficient to eliminate made-up imports or incorrect component APIs. Atlassian combined guidelines with structured code templates:1

  • Fast Template: Optimized for speed, containing 2,000 lines of prototyping guidance covering 8 core component packages.
  • Full Template: Designed for complex interactions, containing 2,500 lines of guidance covering 17 component packages.

These templates preconfigure complex elements like navigation, feature flags, and theme switchers. Furthermore, the template hierarchy is organized to mirror Atlassian's Figma library inheritance model (from core to local product templates). Since designers and developers were already familiar with this structure, it aligned with their existing mental model, reducing setup friction and speeding up onboarding.1

Organizational Scaling & Evals

Teaching AI to use design systems at scale is a socio-technical problem. Atlassian raised organization-wide confidence and adoption through:1

  • Champions Program: Training 6–10% of users as experts who run workshops, Loom walkthroughs, and office hours to support their teams.
  • AI Product Builders Week: A company-wide event where over 1,000 employees paused regular work to build prototypes, producing 115 new ways to use AI and 77 Loom explainer videos.
  • Automated Evals: Recognizing that prompt-tuning and templates are evolving steps, Atlassian's future roadmap focuses on automating evaluations and benchmarking to test design system outputs programmatically, ensuring the system consistently promotes accurate generations.1 See Evals & Observability.
