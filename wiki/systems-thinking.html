Systems Thinking for Design Systems

The mental model beneath the craft: hierarchies, feedback loops, and decomposing a system into solvable concerns.

Thinking in Systems

Frost recommends Donella Meadows' Thinking in Systems as essential reading for understanding how interconnected hierarchies and feedback loops operate in both nature and design systems.1 The relevance is direct: a design system is a network of dependencies where a change at the core propagates outward, so reasoning about it requires thinking in stocks, flows, and feedback rather than in isolated components.

A systems-oriented design method

Nate Baldwin operationalizes systems thinking into a concrete method:2

  • Encapsulation — decompose a system into smaller, independent concerns (responsiveness, accessibility, density) and solve each separately rather than treating the system as one monolith.
  • Parameterization — define elements from constants (base sizes, ratios) rather than hard-coded values, so the system is inherently scalable and dynamic.
  • Archetypes — find commonalities and build a core template, then derive all other components from it instead of designing each individually.
  • Diagrams — use system diagrams to make relationships between tokens, variables, and components legible.

Why decomposition matters

The same instinct to separate concerns recurs across the corpus: separating structure from aesthetics in token architecture, separating size from density in parametric scales, and — most strikingly — separating foundations, style, and behavior so AI agents can reason about smaller "context bubbles."2 Decomposition is the throughline that makes systems both human- and machine-tractable.
