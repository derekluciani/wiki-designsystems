Component Architecture: Web Components & the Layer Cake

How to structure components so they survive framework churn and vary by product without fragmenting.

Web Components as the delivery vehicle

Frost strongly recommends Web Components for delivering a design system across different tech stacks and frameworks, because they bundle HTML, CSS, and JS into a single reusable package that any framework can consume.1 The payoff is reach: one component implementation serves React, Vue, server-rendered pages, and whatever comes next.

The "layer cake"

Frost's compositional model has two tiers:1

  • Basement / base layer — primitive base components shared by everyone.
  • Recipe layer — more complex, composed components (cards, headers) that may legitimately vary by product.

Separating the two lets the shared core stay stable while product teams assemble product-specific "recipes" on top.

Build on native standards

A guiding principle is to build on native web standards wherever possible, because this ensures longevity as specific frameworks come and go.1 The bet is that the platform outlives any given framework.

Keep tokens out of component logic

Frost insists design tokens be kept separate from component logic so they can be themed and updated independently.1 This is the architectural seam that the token architecture page develops in depth.

Perspectives: web-standard reach vs. typed-framework rigor

Frost's web-standards bet optimizes for cross-framework longevity. The type-safety camp optimizes for compile-time correctness inside a typed framework (typically React + TypeScript), accepting framework coupling in exchange for guardrails that forbid invalid prop combinations.2 Both want resilient components; they disagree on whether resilience comes from the platform or from the type system. See Type-Safe Design Systems.
