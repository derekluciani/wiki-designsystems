Modular & Parametric Scales

Generating typography, spacing, and density from base values and ratios — so the system scales by math instead of by hand.

The origin problem

Nate Baldwin's case study starts concretely: his team at Intuit supported button heights of 24/36/48px on an 8-pixel grid. When marketing needed larger buttons, the grid mechanically produced 56 and 64px — but those increments weren't proportional, which is how rigid pixel grids accumulate inconsistency.1

Ditch the pixel grid for modular scales

Baldwin explicitly criticizes rigid pixel grids as non-proportional and prone to inconsistency, advocating modular scales — a base value and a ratio — to generate values instead.1 The same modular formula should drive spacing as well as type, so spacing stays proportional to text size and visual relationships hold.

Responsive typography

Use conditional ratios and base font sizes that adapt to viewport width (small, medium, large, widescreen), and use REM units for text so the system respects the user's browser-level font-size preference — an accessibility requirement, not a nicety.1

Size vs. density

A key distinction Baldwin draws:1

  • Size affects the overall scale of an element, including its text.
  • Density affects only the internal padding of a component.

Separating these dimensions means a density change doesn't break content hierarchy. Padding in component archetypes is defined via relative indices (e.g., −4, −1, 0, 1) rather than hard-coded pixels, making it responsive and adjustable via the scale.

Tooling

Baldwin points to parametric tools: Proportio (his own open-source tool for proportional type/spacing/component systems), Utopia.fyi (CSS clamp() for fluid scaling across viewports), and Token Studio (graph-based, editable representations of system structure).1

Perspectives: parametric generation vs. hand-tuned scales

Baldwin's parametric stance is a clear position in a live debate: should scales be generated from constants and ratios, or hand-picked for each step? His argument is that parameterization makes the system "inherently scalable and dynamic" and is the only approach that survives requests like "we need a bigger button" without breaking proportion.1 It connects directly to the systems-thinking principles of encapsulation and archetypes, and the generated values feed the token architecture.
