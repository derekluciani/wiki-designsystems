AI-Assisted Component Authoring

A real, end-to-end loop for building design-system components with AI — where the speed changes but the quality gates do not.

The Monday.com workflow

Elad Mizrahi describes a concrete ten-step loop for authoring components in Monday's Vibe system with Cursor and Figma MCP.1 The shape:

  1. Design in Figma as usual — but annotate for AI: short, technical, visually explicit (make spacing visible so AI can infer measurements), context-rich.
  2. Generate a component spec via an internal VS Code extension that analyzes the Figma frame and produces a structured technical spec (structure, props, tokens, visual rules); the human reviews and confirms it.
  3. Use Figma MCP to pull screenshots and connect components to "code connect," giving Cursor the visual context to "see" the design.
  4. Load Cursor Rules — persistent .cursor/rules files set up once by engineering, covering structure, styling conventions, Storybook story format, and accessibility (semantic HTML, ARIA, keyboard nav).
  5. Prompt Cursor to generate the component — types, structure, styling, tests — following Monday's conventions. Typical accuracy: 70–90%, higher for simple components.
  6. Generate Storybook stories automatically (props docs, usage, dos/don'ts) inferred from existing patterns.
  7. Update visuals by dropping SVGs/illustrations into Cursor and prompting replacement.
  8. Review and iterate locally — feed Cursor screenshots of actual vs. expected for targeted fixes.
  9. Quality check — push to a branch, open a PR; engineering reviews code quality, a11y, standards, tests; CI runs.
  10. Publish to Vibe, kept in sync across Storybook and Figma so docs aren't maintained twice.

"The speed changes. The quality gates do not."

The defining principle of the workflow: AI compresses the build loop from meetings to minutes, but the human review, accessibility checks, CI, and Figma-side design-system approval all remain.1 This is the operational form of the trust-level discipline — humans keep the merge.

"Mouth coding" and role shift

Frost's complementary framing: this shift lets non-engineers (PMs, sales leads) effectively "code" by prompting agents to assemble interfaces, while engineers retain control through the underlying system's rigor and governance.2 The component-authoring loop is where that promise becomes concrete.

Perspectives: rules and specs as the real product

What makes the Monday loop produce Monday-quality code isn't the model — it's the Cursor Rules and the structured spec.1 This echoes the readiness thesis: the leverage is in the constraint system, not the generator. The same artifacts that make humans productive (clear conventions, good docs, tokens) are what make agents productive — see Agentic Readiness and MCP & the Machine-Readable Interface.
