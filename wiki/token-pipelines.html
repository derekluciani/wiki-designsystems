Token Pipelines: JSON, Style Dictionary & Figma Variables

The plumbing that turns a single token source of truth into platform-specific output — and the naming contract that holds design and code together.

JSON as the tool-agnostic source of truth

Frost recommends storing the design-token source of truth as JSON files so they are tool-agnostic and transformable, then using Style Dictionary (or similar) to transform that JSON into whatever platform-specific output is needed — CSS custom properties, iOS/Android values, React Native tokens.1 One source serves every downstream platform.

The Figma ↔ code naming contract

The token names in Figma (via Figma Variables) must map 1:1 to the names in JSON/code. This shared naming is the contract and API between design and development; misalignment here is the most common source of confusion and breakage.1 The vocabulary becomes shared across designers, developers, QA, CMS systems, and AI tooling.

Typed tokens close the loop

Kathleen McMahon extends the pipeline into the type system: pull variables from Figma through Style Dictionary to generate TypeScript definitions. Typed tokens prevent silent failures and keep multi-brand systems in sync, because a renamed or removed token surfaces as a compile error rather than a runtime surprise.2 This is where the token pipeline meets type-safe systems.

Perspectives: handoff vs. co-creation

Frost is pointed about ownership. Designers have had naming/variable tools (Token Studio, Figma Variables) only since summer 2023 — roughly two years — while developers have named variables for decades. His prescription: developers should not passively accept whatever a designer exports from Figma Variables, but proactively co-create the token architecture, "even if it means picking up tickets not formally assigned to you."1 Co-created systems work vastly better than waterfall handoffs. This reframes a perennial org tension (covered on Adoption, Contribution & Trust) as a technical-collaboration problem, not just a process one.
